Les labels Gmail existants du client — qu'en faire ?

Le client a déjà des labels dans Gmail (Clients, Projets, Admin...). Comment Kyrra les intègre ?

A

Ignorer les labels Gmail

Kyrra crée ses propres labels indépendamment. Les labels Gmail du client restent intacts mais Kyrra ne les utilise pas. Séparation claire.

Pros

  • Simple à implémenter
  • Pas de risque de casser l'organisation existante

Cons

  • Le client se retrouve avec 2 systèmes de labels
  • Kyrra ne bénéficie pas de l'effort d'organisation déjà fait
B

Scanner + proposer un mapping

Kyrra lit les labels Gmail existants et PROPOSE de les intégrer dans sa taxonomie. Ex: "Vous avez un label 'Clients' — voulez-vous que les emails de ce label soient toujours dans 'Important' ?" Le client valide chaque mapping.

Pros

  • Effet "wow" — Kyrra comprend votre organisation
  • Le client part de ses habitudes, pas de zéro
  • L'IA apprend de l'organisation existante

Cons

  • Plus complexe à implémenter (scan labels + IA pour mapping)
  • Besoin d'un scope Gmail API supplémentaire (labels.list)
C

Utiliser les labels Gmail COMME labels Kyrra

Au lieu de créer de nouveaux labels Kyrra, on utilise directement les labels Gmail existants du client + on en crée si besoin. Les classifications Kyrra appliquent les labels que le client connaît déjà.

Pros

  • Zéro friction — pas de double système
  • Le client garde ses repères

Cons

  • Risque de conflits si le client modifie ses labels
  • On perd le "branding" Kyrra dans Gmail
  • Complexe à synchroniser
